waterfall

一开始调研的时候，我们觉得 waterfall 是 weex 最适合用来实现直播首页的组件。于是我们就开干了。

直播卡片布局

由于通栏广告的存在，我们在直播卡片布局这里做了点骚操作：并没有把每张直播卡片作为一个 cell，而是让同一行并排的两张直播卡片放在同一个 cell 里面，这样我们自定义的 waterfall 的 cell 就可以同时兼容直播卡片和通栏广告了。当然，这个骚操作还要对 waterfall 的数据源数组要做二次封装。

安卓 sticky header 不生效问题

waterfall 的子节点 <header> 为 sticky 时，安卓是不生效的。一开始我们在 weex 层做平台区分，iOS 使用 waterfall 的 sticky header，而安卓就自己实现。

后来经过研究，安卓 weex SDK v0.19.0 之后修复了该问题。

性能问题

在我们基本完成 UI 开发，进入自测阶段的时候，我们发现在 iOS 上 waterfall 在快速滑动和加载更多直播卡片的时候，会出现严重卡顿，在高性能机器上面也会卡顿得非常厉害，与原生的性能和体验差距巨大。我们通过 review weex SDK 在 iOS 上的源码，发现 waterfall 组件是基于原生的 collectionView 实现的，它的 cell 本身是重用的，但是 cell 内部的我们使用 weex 布局出来的元素却不是重用的。也就是说，我们往 waterfall 的 cell 里面更新数据，cell 内部原有的视图会被销毁，再重新创建，即便这个 cell 复用前后承载的数据类型是相同。

recycle-list

我们用 waterfall 实现了第一版直播首页之后，测试过程发现了严重的性能问题。于是我们就寄希望于 recycle-list ，希望它能解决 waterfall 上的各种性能问题。然而 recycle-list 却为我们带来了更多的，难以避免的问题。

recycle-list 只能使用<cell-slot>作为其子节点。

cell-slot

<cell-slot> 代表的是列表每一项的模板，它只用来描述模板的结构，并不对应实际的节点。<cell-slot> 的个数只表示模板的种类数，真实列表项的个数是由数据决定的。

个人觉得 recycle-list 的思想和 iOS 中 CollectionView 组件的思想是一样的。CollectionViewCell 复用的时候需要为其绑定一个 cell identification 。其实 weex 的 recycle-list 在 iOS 原生的实现就是基于 CollectionView 的。

好了，步入正题，开始吐槽 <cell-slot> 的坑点：

  • 无法获取内部任何节点的引用。这个坑点的存在，制约了很多后续业务功能的实现，首当其冲的就是，我们无法上报打点信息，80%曝光条件也无法知道，还有就是无法让整个 recycle-list 滚动到指定的位置等等。随便一个就无法让我们继续使用 recycle-list 完成后续开发了。
  • 内部的自定义组件无法单独抽离成一个文件，必须跟 recycle-list 和 <cell-slot> 写在同一个文件里。否则，自定义组件内部的事件是无法回调到 recycle-list 和 <cell-slot> 这一层的。这个虽然能写在同一个文件里面解决，但是对于代码的规范化非常不友好。
  • 至今不能基于 recycle-list 实现一个靠谱的、手感 OK 的下拉刷新功能。

最后我们用回了 waterfall。
